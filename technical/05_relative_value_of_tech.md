# Streamlitひとつで現場が変わった――水資源分野における可視化技術の相対的インパクト

データサイエンティストやMLエンジニアに聞きたい。

直近のプロジェクトで、StreamlitやDash、Gradioあたりで可視化アプリを作ったとき、クライアントの反応はどうだっただろうか。「ああ、ダッシュボードね」「Tableauの方がよくない？」「本番環境ではReactで作り直して」。そんな反応が目に浮かぶ。

インタラクティブな可視化は、いまやデータサイエンスプロジェクトの標準装備であり、それ自体が驚きを生むことは少ない。BIツールの成熟、ノーコードプラットフォームの台頭により、「データを可視化する」という行為のコモディティ化が進んでいる。

だが、この「コモディティ化した技術」を、ザンビアの水資源管理の現場に持ち込んだとき、まったく予想しなかった反応が返ってきた。相手の技術者たちが、目の色を変えたのだ。

技術の価値は相対的である。この当たり前のことを、われわれは身をもって思い知らされた。

## ザンビアの水資源管理における技術スタック：2024年の現実

まず、コンテキストを共有しておく。

ザンビアの水資源・土木分野における「技術スタック」は、端的に言えばこうだ。

- **データ収集**: 住み込み観測員による手動読み取り（朝・昼・晩の水位目視観測）→ 紙の観測記録簿に手書き記録
- **データ管理**: 観測記録の手入力によるExcelスプレッドシート化、または紙台帳のまま保管
- **データ分析**: Excel上での集計・グラフ作成、印刷した表の手作業での突き合わせ
- **意思決定支援**: 紙の地図への手書き注記、印刷資料の配布と対面会議

GIS（地理情報システム）については、WARMA（Water Resources Management Authority）やZESCOの一部技術者がQGIS等を扱えるものの、JICAや世界銀行の技術協力プロジェクトを通じて導入された経緯が多く、日常的な業務プロセスに統合されているわけではない。データベースの概念はあっても、リレーショナルDBが運用されているケースは限定的だ。

要するに、データパイプラインの全工程がアナログまたはセミアナログであり、「データの収集から意思決定までのリードタイム」が極めて長い。

一方、日本のデータサイエンスプロジェクトでは、以下のようなスタックが「最低限」として期待される。

- データ取得のAPI化・自動パイプライン（Airflow、Prefect等）
- クラウドDWH（BigQuery、Redshift等）でのデータ管理
- BIツール（Tableau、Looker、Power BI等）または自作ダッシュボード
- CI/CDを含む開発・デプロイフロー

この二つの世界のギャップが、どれほど大きいか。同じ「水資源の可視化」という課題に取り組んでいても、スタート地点がまるで違う。

## 実装の詳細：何を、どう作ったか

われわれがザンビア側に提供したプロトタイプの技術的な概要を述べる。

**フレームワーク**: Python + Streamlit（Dash等も候補であったが、Streamlitのラピッドプロトタイピング特性と低い学習コストからこれを選定）

**主要機能**:
1. **地理空間可視化**: 河川流量の観測地点を地図上にプロット。Folium（Leaflet.jsのPythonラッパー）またはMapbox GLを用いたインタラクティブマップ
2. **時系列データの動的表示**: 観測地点の選択に連動して、流量・水位の時系列グラフをPlotlyで描画。期間のスライダー操作でズームイン・アウト
3. **空間的フィルタリング**: 流域・行政区画単位でのデータ絞り込み
4. **比較機能**: 複数地点の時系列データを重ね合わせ表示

技術的には、以下のようなごく標準的な構成だ。

```
data source (CSV/Excel)
  → pandas DataFrame
    → Streamlit widgets (selectbox, slider, multiselect)
      → Plotly/Folium による描画
```

ETL処理は最小限。データソースはCSVかExcelファイルの直接読み込み。データベースすら不要。認証・認可もなし。デプロイも、ローカル実行またはStreamlit Cloud等の無料ティアで十分だ。

開発工数でいえば、経験のあるデータサイエンティストなら数日、長くても1-2週間で構築可能なレベルのプロトタイプだ。

**日本の感覚では、これはデモ用のモックアップに過ぎない。**

## 現場でのインパクト：定性的な観察

このプロトタイプをザンビアの水資源関係者に提示した際の反応を、定性的に整理する。

**即時的な変化:**
- 紙資料を参照する「ページめくり」が消え、全員がスクリーンを注視する状態に移行
- 「この地点のデータを見たい」→ クリック → 即座にグラフ表示、というサイクルにより、会議中のデータ参照速度が桁違いに向上
- 「もしAの地点とBの地点を比較したら？」という仮説検証型の議論がリアルタイムで可能に（従来は次回会議まで持ち越し）

**構造的な変化の兆し:**
- 「このアプリに、我々の持っている過去のデータも入れられるか？」という、データ統合への意欲
- 「関係省庁にもこの画面を見せたい」という、情報共有手段としての認識
- 「リアルタイムの観測データを自動で反映できないか？」という、データパイプライン構築への発展的要求

注目すべきは、**ユーザーが自発的に「次のステップ」を構想し始めた**ことだ。技術受容モデル（TAM: Technology Acceptance Model; Davis, F.D., "Perceived Usefulness, Perceived Ease of Use, and User Acceptance of Information Technology," *MIS Quarterly*, Vol. 13, No. 3, pp. 319-340, 1989）でいう「知覚された有用性（Perceived Usefulness）」と「知覚された使いやすさ（Perceived Ease of Use）」の両方が、一つのプロトタイプで同時に満たされた格好だ。

## 技術のアービトラージ：なぜこれが起きるのか

ここで、この現象を構造的に捉えてみたい。

ある技術のインパクトは、以下の式で近似できる。

```
技術のインパクト ≈ (技術が提供する能力) - (現状の能力)
                = Δ（デルタ：差分）
```

日本のデータサイエンス市場では:
- 現状の能力: BIツール、自動パイプライン、クラウドインフラ（高い）
- インタラクティブ可視化の提供能力: 同程度（高い）
- **Δ ≈ 小**

ザンビアの水資源分野では:
- 現状の能力: 紙、Excel、手作業（低い）
- インタラクティブ可視化の提供能力: 同程度（高い）
- **Δ ≈ 極めて大きい**

同じ技術でも、適用先の「ベースライン」が異なれば、生じるΔは劇的に変わる。これは経済学における裁定取引（アービトラージ）と同型の構造だ。ある市場で「均衡価格」に達した技術が、別の市場では「均衡」に程遠い状態にある。このギャップにこそ、インパクトの源泉がある。

**技術の価値は絶対的ではなく、適用先のコンテキストとの相対関係で決まる。**

これは言われてみれば自明だが、技術者はしばしば忘れる。われわれは「より高度な技術」「より新しいフレームワーク」を追いかけることに熱心だが、「既存の技術を最もΔが大きい場所に届ける」という方向の最適化は、見落としがちだ。

## 技術選定と実装における考慮点

この知見を実務に落とし込むと、新興国の水資源・インフラ分野向けのプロダクト開発において、以下の技術的考慮が重要になる。

**1. オフライン耐性の設計**
ザンビアのインターネット普及率は人口の約30%程度にとどまり（ITU, 2023; World Bank DataBank）、モバイルSIM普及率は90%超であるものの、3G/4Gカバレッジは都市部に集中し、農村部では通信環境が極めて不安定だ（ZICTA Annual Report）。常時接続を前提としたSaaSモデルは機能しない場面が多い。PWA（Progressive Web App）化やローカルキャッシュの活用、さらには完全オフラインで動作するElectronベースのデスクトップアプリも選択肢に入る。

**2. 低スペックデバイスへの最適化**
クライアント側のデバイスは、スペックが限られたラップトップや、場合によってはスマートフォンだ。重いJavaScriptバンドルや大量のDOM操作は避け、サーバーサイドレンダリングやデータの段階的読み込みを検討すべきだ。StreamlitやPanel等のPythonベースフレームワークは、この点で相性が良い（レンダリングの大部分がサーバーサイドで完結するため）。

**3. データフォーマットの柔軟性**
現場のデータは、CSV、Excel、場合によってはスキャンされた紙資料のPDFで存在する。多様なフォーマットに対応したデータインジェスション層が必要だ。OCRパイプラインの組み込みも、中長期的には検討に値する。

**4. 段階的な複雑性の導入**
最初からフルスタックのデータプラットフォームを構築しようとするのは失敗パターンだ。まずはCSV読み込み → 可視化というミニマムなパイプラインで価値を証明し、ユーザーの要求に応じてデータベース化、API化、自動更新と段階的に進化させるアプローチが現実的だ。

## 技術者としての問い直し

最後に、技術者としての内省を一つ。

われわれは「技術的にチャレンジングな問題」に惹かれやすい。LLMのファインチューニング、分散システムのスケーリング、リアルタイムストリーミング処理。それ自体は健全な知的好奇心だ。

だが、ザンビアの現場で目の色を変えた技術者たちの顔を見て、思った。**技術的には「つまらない」かもしれない仕事の中に、社会的には最も大きなインパクトを生む仕事が隠れていることがある。**

「Streamlitでダッシュボード作りました」。技術者のコミュニティでは、いまさら誰も注目しない話だ。でも、それが国家の水資源管理の意思決定を変え、ひいてはエネルギー政策に影響を与えうるとしたら。そのインパクトは、最先端のMLパイプラインを構築するよりも、はるかに大きいかもしれない。

**技術の価値は相対的だ。そして、最もΔが大きい場所に技術を届けられる技術者こそが、最も大きなインパクトを生み出せる。**

この事実を、コードを書くすべての人に伝えたい。あなたが「技術的につまらない」と切り捨てたその仕事が、世界のどこかで最も必要とされている仕事かもしれないのだから。
